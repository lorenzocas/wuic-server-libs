Sicurezza

🔑 Migrazione hashing password da SHA1 a PBKDF2 Alta
appsettings.json dichiara encriptionMethod: SHA1. SHA1 è deprecato per le password. Migrare a PBKDF2 (o bcrypt) con salt per utente.
S Todo NET
▼
NET KonvergenceCore / CrmApp appsettings
  • Cambiare "encriptionMethod": "SHA1" in "encriptionMethod": "PBKDF2" in appsettings.json.
  • Verificare che MetaService.cs (o il metodo di autenticazione) supporti già PBKDF2: cercare nel sorgente il dispatch su encriptionMethod. Se non presente, aggiungere il branch PBKDF2 usando System.Security.Cryptography.Rfc2898DeriveBytes.
  • Aggiungere colonna password_salt nvarchar(64) alla tabella utenti su MetadataCRM se non già presente.
  • Creare una migration script SQL che re-hash le password esistenti (opzionale: forzare cambio password al primo login con il nuovo algoritmo).
⚠️ Priorità di sicurezza. SHA1 senza salt è vulnerabile ad attacchi rainbow table. Questo intervento è indipendente da tutte le altre feature e non richiede modifiche al frontend.

🛡️ Rate limiting sull'API CrmActions Media
L'endpoint POST /api/CrmActions/execute/{actionKey} non ha protezione contro chiamate abusive. Limitare a N richieste per minuto per utente/IP.
S Todo NET
▼
NET CrmApp Program.cs
  • Aggiungere il pacchetto NuGet Microsoft.AspNetCore.RateLimiting (incluso in .NET 7+, nessun pacchetto esterno necessario).
  • In Program.cs di CrmApp: builder.Services.AddRateLimiter(o => o.AddFixedWindowLimiter("crm-actions", opt => { opt.PermitLimit = 30; opt.Window = TimeSpan.FromMinutes(1); opt.QueueLimit = 5; })).
  • Applicare il limiter a CrmActionsController con [EnableRateLimiting("crm-actions")] o globalmente su tutte le route API.
  • Aggiungere app.UseRateLimiter() nella pipeline middleware prima di app.MapControllers().
